# 📋 自动化代码审查框架

## 🎯 审查维度与优先级

| 优先级 | 维度 | 检查项 | 阻断合并 | 自动修复 |
|--------|------|--------|----------|----------|
| 🔴 Critical | 安全 | SQL 注入/XSS/凭证泄露/权限绕过 | ✅ 是 | ⚠️ 建议 |
| 🔴 Critical | 正确性 | 空指针/越界/竞态/未处理异常 | ✅ 是 | ✅ 可尝试 |
| 🟡 Warning | 性能 | N+1 查询/内存泄漏/冗余计算 | ⚠️ 建议 | ✅ 可尝试 |
| 🟡 Warning | 可维护性 | 重复代码/复杂度过高/命名模糊 | ⚠️ 建议 | ✅ 可尝试 |
| 🔵 Info | 风格 | 格式/注释/文档/日志规范 | ❌ 否 | ✅ 自动 |

## 📤 结构化输出格式

```markdown
## Code Review: [文件/PR/功能]

### 📊 摘要
> [1-2 句] 变更概述 + 整体质量评估 + 关键风险提示  
> 示例："实现用户登录，核心逻辑正确，但发现 2 个安全风险需修复"

### 🔴 必须修复 (Critical)
| # | 位置 | 问题 | 风险 | 修复建议 | 自动修复 |
|---|------|------|------|----------|----------|
| 1 | src/auth:42 | SQL 注入风险 | 数据泄露 | 使用参数化查询 | ✅ [代码] |
| 2 | src/utils:18 | 空指针可能 | 服务崩溃 | 添加空值检查 | ✅ [代码] |

### 🟡 建议优化 (Warning)
| # | 位置 | 问题 | 影响 | 优化建议 | 自动修复 |
|---|------|------|------|----------|----------|
| 1 | src/api:55 | N+1 查询 | 响应慢 | 添加 select_related | ✅ [代码] |
| 2 | src/config:12 | 硬编码配置 | 维护难 | 移至环境变量 | ⚠️ 指导 |

### ✅ 做得好的地方
- [ ] 清晰的错误处理逻辑
- [ ] 完善的边界条件测试
- [ ] 符合项目编码规范

### 🎯 结论与建议
🟡 请求修改（2 个 Critical 问题需修复后合并）

📋 下一步:
1. 修复 🔴 问题 #1, #2
2. 考虑优化 🟡 问题 #1
3. 重新运行测试验证
```

## 🤖 智能触发规则

### 自动触发场景
- 用户说"审查这段代码" / "这个实现有问题吗" / "帮我看下"
- 检测到代码包含：数据库操作/网络请求/文件读写/认证逻辑
- 用户提到关键词：安全、性能、bug、优化、重构

### 手动触发方式
```bash
# 显式调用
/skills code-review-automation
"请审查：@src/auth/login.ts"

# 指定重点
"重点检查安全方面：@src/api/"
"忽略格式问题，只查逻辑风险"
```

## 🛠️ 自动修复能力

### 可自动修复的问题
```
✅ 代码格式：Prettier/ESLint 自动修复
✅ 简单重构：提取函数/重命名变量/简化条件
✅ 安全修复：参数化查询/空值检查/输出转义
✅ 测试补充：添加缺失的边界用例
```

### 修复输出格式
```typescript
// 🔧 自动修复建议：src/auth/login.ts:42
// 问题：SQL 注入风险
// 修复：使用参数化查询

// Before ❌
db.query(`SELECT * FROM users WHERE email = '${email}'`)

// After ✅  
db.query('SELECT * FROM users WHERE email = ?', [email])

// 验证：运行 `npm run test:security` 确认无注入风险
```

## 🧪 示例

### 用户请求
```
"审查这个登录实现：@src/auth/login.ts"
```

### 技能响应
```markdown
## Code Review: src/auth/login.ts

### 📊 摘要
> 实现邮箱 + 密码登录，核心流程正确，但发现 1 个安全风险 (🔴) 和 2 个优化点 (🟡)

### 🔴 必须修复
| # | 位置 | 问题 | 风险 | 修复建议 |
|---|------|------|------|----------|
| 1 | L42 | SQL 查询字符串拼接 | 注入攻击 | 改用参数化查询 |

### 🟡 建议优化
| # | 位置 | 问题 | 影响 | 优化建议 |
|---|------|------|------|----------|
| 1 | L18 | 日志记录用户邮箱 | 隐私泄露 | 脱敏处理：`user***@domain` |
| 2 | L55 | 未限制登录尝试频率 | 暴破风险 | 添加 Redis 计数 + 锁定 |

### ✅ 做得好的地方
- ✓ 密码使用 bcrypt 加密 (cost=12)
- ✓ 错误消息不泄露"用户不存在"细节
- ✓ 添加了基础输入验证

### 🔧 一键修复
```typescript
// 🔧 修复 L42: SQL 注入
// Before
const query = `SELECT * FROM users WHERE email = '${email}' AND password = '${hash}'`

// After ✅
const query = 'SELECT * FROM users WHERE email = ? AND password = ?'
const result = await db.query(query, [email, hash])
```

### 🎯 结论
🟡 请求修改（修复🔴#1 后可合并）

📋 建议顺序:
1. 应用 🔧 修复 #1 (安全关键)
2. 考虑 🟡 #2 (隐私保护)
3. 规划 🟡 #3 (防爆破，可后续迭代)
```
